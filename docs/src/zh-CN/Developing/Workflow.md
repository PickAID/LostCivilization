# 工作流 {#workflow}

本页定义项目的固定推进顺序。我们不按“想到什么就先做什么”的方式推进，而是按能否证明主循环、能否稳定复现、能否被文档收口三个标准推进。

```mermaid
flowchart LR
    Slice["主循环竖切片"] --> Shell["单一文明外壳样板"]
    Shell --> Resonance["共鸣与构筑深化"]
    Resonance --> Expansion["表现与内容扩张"]
```

## 固定顺序 {#fixed-order}

项目按下面顺序推进：

| 阶段 | 目标 | 通过条件 |
| --- | --- | --- |
| 主循环竖切片 | 跑通前期发现 -> 正式勘探 -> 激活 -> 现场运行 -> 回收 | 玩家能完整完成一轮遗址行动 |
| 单一文明外壳样板 | 证明同一主循环可以承载明确的文明身份 | 线索、激活语义、压力和回收开始出现稳定差异 |
| 共鸣与构筑深化 | 让遗物与现场互动真正改变处理方式 | 同一现场因为构筑不同而出现可复现的决策差异 |
| 表现与内容扩张 | 在已成立的循环上增加变体和密度 | 新内容是在加厚核心，不是在遮掩核心 |

这条顺序不能反过来。主循环没有站住之前，文明外壳只会变成包装；共鸣没有站住之前，内容扩张只会变成堆量。

## 试玩门槛 {#playtest-gate}

每一轮阶段推进都要先过试玩门槛。第一版最重要的门槛如下：

1. 一轮遗址流程应当能在大约十到二十分钟内完整走完。
2. 试玩后，测试者应当能明确说出自己经历了哪些阶段，而不是只记得零散功能。
3. 流程里至少要能辨认出一个失败点、一个回收点和一个继续投入的理由。
4. 如果试玩反馈仍然集中在“这是什么”“该做什么”“为什么这样算完成”，内容扩张就要暂停。

试玩的目的不是证明内容多，而是证明主循环清楚、可读、可重复。

## 文档同步规则 {#documentation-sync-rules}

文档、包内容和代码实现必须同轮收口，不允许长期脱节。

| 变化类型 | 必须更新什么 |
| --- | --- |
| 设计边界变化 | 对应 `Design` 页面 |
| 对象所有权、数据结构或运行时边界变化 | 对应 `ModdingDeveloping/Implementation` 或流程页 |
| pack 侧装配方式、模组职责或资源组织变化 | 对应 `Modpacking` 页面 |
| 已交付行为对外语义发生变化 | `Changelog` |

如果实现已经改了，而永久文档还没同步，这项工作就不能算完成。聊天记录、临时草稿和任务拆分文件都不能替代永久页面。

## 推进纪律 {#delivery-discipline}

推进时遵守三条纪律：

1. 每次只推进一个主要矛盾。当前主矛盾如果是主循环不清楚，就不要同时扩文明、扩表现和扩模组。
2. 任何新增内容都要回答“它属于哪个阶段、服务哪个对象、通过哪个门槛”。
3. 只要某项工作改写了项目真相，就必须同步到对应永久页面，而不是等下一轮再补。

## 禁止项 {#prohibited-patterns}

- 主循环还没有稳定，就提前铺开多文明、多遗址类型或大规模内容包。
- 关键判断只留在聊天、临时计划或一次性草稿里。
- 用更多文本、更多 lore 或更多展示页面去掩盖规则仍然模糊。
- 因为实现方便，就让 docs、pack 和 runtime 的边界重新混在一起。
